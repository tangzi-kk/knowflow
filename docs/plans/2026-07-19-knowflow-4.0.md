# KnowFlow 4.0 Implementation Plan

> **For Codex:** Use `subagent-driven-development` to implement this plan task-by-task. Every behavior change follows RED-GREEN-REFACTOR, then separate specification and quality reviews.

**Goal:** Deliver a source-reproducible KnowFlow 4.0 whose cross-end sync operations are safe, idempotent, observable, recoverable, upgradeable, and never report success without a proven write result.

**Architecture:** Keep the existing localhost HTTP + `obsidian://` wake-up architecture. Add only the smallest state seams required to make it reliable: one Obsidian sync state store/coordinator, one browser storage migration module, one browser workflow state module, and one toolbar reducer. Use bounded JSON state rather than a database, Promise locks rather than a queue, and HTTP results rather than a callback service.

**Tech Stack:** TypeScript 5.9, Node.js built-in test runner/assertions, esbuild, Obsidian desktop API, Chrome/Edge Manifest V3, existing `@sync/shared` workspace.

## Release 3.4.0 — Conflict and Recovery Safety

### Task 9: Implement three-way sync decisions

**Files:**
- Create: `packages/ob-plugin/src/syncDecision.ts`
- Create: `packages/ob-plugin/tests/syncDecision.test.mjs`
- Modify: fetch and pushback handlers.

Decision inputs are `baseHash`, `localHash`, and `remoteHash`. Both changed and unequal means `CONFLICT`; missing reliable base means pause. No AI merge and no automatic winner.

### Task 10: Add bounded pre-write recovery snapshots

**Files:**
- Create: `packages/ob-plugin/src/recovery.ts`
- Create: `packages/ob-plugin/tests/recovery.test.mjs`

Keep a small rotating set of local snapshots before destructive writes. Failure to snapshot aborts the write; failure after remote write records a repair-needed terminal state.

## Release 4.0.0 — Product Closure

### Task 16: Consolidate the sync center and remove proven-dead compatibility code

Use one Obsidian sync center for health, conflicts, pending deletion, recent failure, and index rebuild. Keep browser Side Panel limited to capture confirmation, final result, and recent activity. Remove compatibility code only when migration and rollback tests prove it unused.

### Task 17: Run the release audit

Automated and real-environment gates must cover fresh install; every intermediate upgrade; 3.2.1 upgrade; rollback; pull; pushback; simultaneous edits; duplicate requests; 20-way concurrency; bad token; port collision; missing/expired CLI; 429; timeout; malformed JSON/YAML; huge document; image failure; plugin unload; browser restart; killed MV3 worker; corrupt state; and secret scanning.

For every failure, assert all three: no data loss, no false success, and a recoverable next action.

### Task 18: Produce traceable release artifacts

Update manifests, package versions, `README.md`, `VERSION_INDEX.md`, `CHANGELOG.md`, `PROGRESS.md`, migration guide, and SHA256 manifests. Every artifact must identify the same version and commit. Do not push or deploy without explicit user confirmation.
